# 九点五、Sable 的 Pose3d.scale：API 设计在先，物理实现烂尾

### 背景
`Pose3d`（Sable Companion 中的位姿类）包含三个字段：`position`（位置）、`orientation`（四元数旋转）、`scale`（缩放）。Java Javadoc 明确说 SubLevel 是一个 *"Isometry, or a 3D pose with translation, rotation, and scale"*。

然而 `scale` 字段在**渲染层是有效的**（`SubLevelRenderData.getTransformation()` 使用它），但在 **Rust Rapier 物理引擎中从未被实现**。

### 上游代码考古结论

| 层 | 缩放状态 |
|----|---------|
| Java API（`Pose3d`） | ✅ `scale` 字段存在，渲染代码使用它 |
| Java JNI（`Rapier3D.createSubLevel`） | ❌ 只传 7 个 double（位置+四元数） |
| Rust JNI（`lib.rs`） | ❌ 未解析 scale 参数 |
| Rust 粗筛 AABB（`collider.rs`） | ❌ 未应用 scale |
| Rust 窄相碰撞（`dispatcher.rs`） | ❌ 未缩放碰撞盒 |
| `KinematicContraption` | ⚠️ 显式将 scale 重置为 identity |

### 结论
缩放功能是上游**刻意设计但未完成**的功能。Java API 层为将来预留了接口，但 Rust 物理引擎从未接入。这不是 bug，是半成品。

### 跨语言类型注意事项
如果未来需要重新实现，注意 Rust 侧的类型匹配：
- `voxel_collider_data.collision_boxes` 类型为 `Vec<(f32, f32, f32, f32, f32, f32)>`
- scale 变量为 `NaVector3<f64>`（Rapier 的 `Vector3<f64>`）
- 两者相乘需显式 `as f64` 转换
