# SubLevel 缩放考古记

> ——— 2026-06-05，一场关于"上游烂尾功能"的技术考古始末

---

## 楔子：比例耦合

CrossOut 的网格精度是 Minecraft 的 3 倍。这意味着在 Crossout 中建造的车辆若要在 Minecraft 中以 1:1 的对应关系再现，需要将整个 SubLevel 缩小到 1/3。

这个需求被称为 **"比例耦合"**——不是什么复杂的物理问题，只是一个简单的缩放变换。理论上，给 `Pose3d.scale` 设个 `(1/3, 1/3, 1/3)` 就够了。

---

## 第一幕：发现

我们翻开了 Sable Companion 的 `Pose3d` 源码：

```java
public class Pose3d {
    public Vector3d position;
    public Quaterniond orientation;
    public Vector3d scale;   // ← 这个字段存在，但从未被使用
}
```

而 `SubLevel` 的 Javadoc 写着：

> *"An Isometry, or a 3D pose with translation, rotation, and **scale**."*

渲染代码也确实在用这个字段（`SubLevelRenderData.getTransformation()` 调用 `store.scale()`）。一切都指向一个结论：**缩放功能已经做好了，只需要开启即可。**

我们以为这只是一次简单的配置修改。

---

## 第二幕：深入

实际走了一遍数据流才发现，这个"已做好"的假象有多脆弱：

```
设置 pose.scale(1/3)
  → Java 侧：✅ 渲染层识别了（座舱高度/悬挂位置变了）
  → JNI 桥接：❌ 只传了 7 个 double（位置+四元数），没有 scale
  → Rust 物理引擎：❌ 从未接收 scale 参数
  → 粗筛 AABB：❌ 没缩放
  → 窄相碰撞：❌ 没缩放
  → 最终结果：车在空中悬浮，碰撞完全错位
```

我们面对的不是"开关没开"，而是一整条从 Java 到 Rust 再到 Rapier 碰撞检测的**全链路断层**。

---

## 第三幕：修补

我们花了整整一天做了这些修改：

| 文件 | 修改内容 | 语言 |
|------|---------|------|
| `lib.rs` | `ActiveLevelColliderInfo` 新增 `scale` 字段；`createSubLevel` JNI 参数 7→10；新增 `setScale` JNI | Rust |
| `collider.rs` | `compute_local_aabb` 应用缩放（粗筛） | Rust |
| `dispatcher.rs` | 窄相碰撞的碰撞盒中心+半边长乘以缩放系数 | Rust |
| `Rapier3D.java` | 更新 native 方法签名 | Java |
| `RapierPhysicsPipeline.java` | `add`/`readPose`/`teleport` 传递缩放 | Java |
| `CrossAndOutConfig.java` | 新增 `SUBLEVEL_SCALE_ENABLED` + 缩放常数 | Java |
| `ServerMountHandler.java` | 上车时设置 `pose.scale()` | Java |
| `build-rust.yml` | 添加 Rust 构建 CI | YAML |
| `gametests.yml` | 添加 `upload-artifact` + `workflow_dispatch` | YAML |

CI 构建经历了 **3 次失败**才成功：
1. Rust 借用冲突 → `{ }` 作用域块修复
2. 数组大小不匹配（7→10）→ 修复
3. f32/f64 类型不匹配 → `as f64` 显式转换修复

---

## 第四幕：结果

实际测试结果令人沮丧：**没什么变化。**

- ✅ 缩放参数成功传入引擎日志（`Set SubLevel ... scale to (0.333, 0.333, 0.333)`）
- ✅ 座舱动力系统正常工作
- ✅ 无崩溃、无报错
- ❌ 车辆视觉大小没有变化
- ❌ 碰撞行为没有变化

我们补全了 Java→JNI→Rust→粗筛→窄相的全链路，但最终效果不可感知。

---

## 复盘：为什么失败？

### 技术层面
我们可能只解决了**一部分**物理问题。SubLevel 的缩放涉及：

- ✅ 物理引擎内部的碰撞检测（我们改了）
- ❓ 实体与 SubLevel 的碰撞（`SubLevelEntityCollision.java`，我们没动）
- ❓ 渲染管线的缩放应用（Java 侧已有，但实际没生效？）
- ❓ 网络同步的缩放传递
- ❓ 质量/惯性张量的缩放
- ❓ 约束（弹簧/悬挂）的缩放适配

我们只挖到了冰山一角。

### 策略层面

这不是一个"需要修复的 bug"，而是一个**上游刻意设计但未完成的功能**。从代码考古来看：

- `Pose3d.scale` 在 Java 渲染层是有效且被使用的
- `KinematicContraption.sable$getLocalPose()` 显式将 scale 设为 identity
- Rust 侧的 `createSubLevel` 从未接收 scale 参数
- 上游（ryanhcode/sable）目前没有开发此功能的迹象

RyanH 设计 API 时预留了缩放接口，但可能因为当时没有需求（也没有衍生作品需要它），物理引擎的实现就被搁置了。

### 经济学层面

> *"用胶水把一个饼干渣修理成完整的一块饼干，和拿着刹车片和老板说把车修完整没什么区别。"*

我们在试图补全一个上游自己都没完成的底层功能。这需要深入理解 Sable 的整个物理架构——从 JNI 边界到 Rapier 的碰撞调度——而这对于我们（IAC-P 开发者）来说，属于"物理引擎底层设计"的范畴，与"在 Minecraft 中实现 Crossout 玩法"的目标越来越远。

---

## 转机可能在什么时候？

### 上游方案（最理想）

ryanhcode/sable 未来某天正式实现 `Pose3d.scale → Rapier` 的原生支持。届时我们只需：
1. 同步上游代码
2. 将 `SUBLEVEL_SCALE_ENABLED = true`

但从上游目前的活跃度和代码状态来看，这个时间点**不可预期**。

### 轻量替代方案（更现实）

如果未来的 IAC-P 开发中确实需要缩放能力（不只是比例耦合），可以考虑**不依赖 Sable 的物理缩放**，而是在 IAC-P 的逻辑层做等比例坐标换算：
- 在 `ServerMountHandler` 中自行维护一套缩放后的坐标映射
- 不修改 SubLevel 的物理 size，只在逻辑层换算输入的坐标/速度/力

这样可以绕过 Sable 的 Rust 侧，完全在 Java 层解决问题。但缺点是需要手动管理更多映射关系。

### 搁置（当前决策）

缩放功能暂停。`SUBLEVEL_SCALE_ENABLED = false`（默认关闭）。所有修改保留在 `HaiNaBaiChuan75/sable` 的 `feature/sublevel-scale` 分支上，供未来参考。

---

## 路线抉择：不再等待

当天晚些时候，我们在回顾整个缩放考古过程后，做了一个更根本的决策：

### 问题
缩放功能的有无，让我们对 IAC-P 的路线陷入两难：

| 路线 | 优点 | 代价 |
|------|------|------|
| **Crossout 高精度路线** | 蓝图精细，比例接近原作 | 需要等待缩放，且需要**一套完整的多方块建造 UI**（预览、旋转、批量放置）——这是编辑器级别的工程，远超出模组开发的范畴 |
| **Blocky car 低精度路线** | 现有技术栈即可实现，一个月内可玩 | 格子感重，比例不像 Crossout |

### 关键洞察：多方块建造不是"加点功能"的事

要走 Crossout 的高精度路线，==不只是解决了缩放就能跑==。高精度网格意味着：
- 玩家需要**在 1/3 格粒度上放置和预览方块**
- 需要**旋转和镜像工具**
- 需要**蓝图保存/加载系统**

这套玩意本质上是一个**独立的建造编辑器**。以模组开发的人力，造它比造物理引擎更不现实。

反观 Blocky car 路线：
- Create 的扳手+装配就是建造系统
- 已有的悬挂/座舱/摩擦圆直接能用
- 我们只需要做部件损坏、WASD 映射这些"锦上添花"的功能

### 最终决定

> **不再为一个不存在的功能等待。走 Blocky car 路线，先拿出可玩的成品。**

- 不是"等缩放来了再适配"，而是 **"这就是我们要做的游戏"**
- 物理参数以 Minecraft 格子为基准，兼容即可
- 如果未来缩放真的来了，蓝图导出时做后处理变换即可，不影响运行时
- 已经和朋友们做过期望管理——"我只是自己用爱发电做点代餐吃，别把我当救世主"

这个决策的核心：**宁可要一个可玩的方块车，也不要一个永远在"等缩放"的完美计划。**
