# 文档体系评估与改进建议

> 本文档是对整个项目文档体系（`《中控载具工坊：范式》项目推进文档 3.0`）的**系统性审视**。
> 列出当前文档的优势、缺口，以及按优先级排序的补充建议。
>
> 本文档本身不直接修改其他文档，而是作为改进计划的索引和讨论起点。

---

## 一、总体评价

经过全面阅读，本项目的文档体系在 Minecraft 模组开源项目中属于**顶尖水平**。核心优点：

### 1.1 元规则先行

`0.1-文档更新规则.md` 定义了文档管理的"宪法"：
- **单一事实来源** — 每一条信息只在一个地方维护
- **文学性与技术性分离** — `7-开发日志/` 记录过程，`5-技术参考/` 沉淀知识
- **变更同步更新** — 代码变更对应文档必须同步

绝大多数开源项目忽略"文档的文档"，这导致文档体系随项目膨胀而腐烂。本项目在这方面的制度设计非常出色。

### 1.2 知识管理成熟

`5.1-关键技术要点/` 的 29 篇文档 + `5.2-踩坑快速索引.md` 的二维索引（按症状 + 按组件），构成了一个可维护、可搜索的项目知识库。

每条踩坑记录格式统一（背景 → 根因 → 解决方案 → 参考），这是专业级知识管理的标志。

### 1.3 Bug 追踪有闭环

`4.1-当前状态.md §已修复 Bug` 不仅记录修复了什么，还记录了**根因**和**修复方案**。这对未来排查回归问题极有价值——新协作者看到某个 Bug 的症状时，可以快速判断是不是老问题复发。

### 1.4 设计哲学有约束力

`5.4-设计哲学与路线图/1-战略罗盘与设计哲学.md` 的 8 条原则不是空话，而是实际指导了关键决策：
- "代码做胶水，不造轮子" → 放弃抽象齿轮模式，拥抱位置模式 PD 伺服
- "美术占位符主义" → 不做官方素材包，做社区导入框架
- "渐进式复杂度" → 10 秒开走 vs 10 年调不完的两层架构

### 1.5 文档与代码一致性好

`2.1-功能总表.md` 的 79+ 项功能与 `3.3-代码文件索引.md` 的 ~40 个文件对应清晰，命名规范统一。这是文档体系健康与否的最直接指标。

## 二、文档缺口与补充建议

以下按优先级从高到低排列，每项包含：**问题描述 → 建议内容 → 推荐位置**。

### 🔴 高优先级

#### 2.1 测试策略文档

**问题**：目前没有专门描述测试策略的文档。`4.1` 中的"已验证的代码行为"列表（~60 条）记录了代码走查和实测验证的结果，但缺少：
- 测试的整体架构（单元测试 vs 集成测试 vs 手动测试）
- 测试框架和运行方式
- 每次发布前需验证的核心场景清单
- 物理参数变更后的回归验证方法
- 多玩家场景的测试覆盖

**建议内容**：

```
# 测试策略与验证方法

## 测试层级
- 单元测试：纯函数测试（EngineModel、TransmissionModel、BrushTireModel 等）
  - 运行方式：./gradlew test
  - 覆盖率目标：XXX
- 集成测试：SubLevel 装配/拆卸、约束链、网络包编解码
  - 运行方式：./gradlew runGameTestServer
  - 覆盖场景：XXX
- 手动测试：每次发布前必测的核心场景
  - 上车/下车/换挡/转向/开火/炮塔追踪
  - 边界条件：断线重连、死亡、传送、多人同乘

## 物理参数验证方法
- 修改编译时常量后需要检查的行为清单
- 预期极速/加速度/抓地力范围的速查表

## Bug 回归测试
- 已修复 Bug 的验证方法（参考 4.1-当前状态.md §已修复 Bug）
- 回归检查清单
```

**推荐位置**：`5-技术参考/测试策略与验证方法.md` 或 `3-架构与文件结构/测试策略.md`

**预计工作量**：中（1-2 天，含清单整理）

#### 2.3 已知限制汇总

**问题**：`4.1-当前状态.md` 中分散着大量已知限制：
- "重量仅上车时同步一次"（低优先级）
- "客户端无 MassData API"（低优先级）
- "physicsTick 线程安全"（隐患）
- "双模式下车坐标系不一致"（中优先级待修复）
- "多人摄像机同步"（待验证）
- "锚点更新后约束未跟随"（低优先级）
- "轮胎款式物理属性未区分"（设计缺口）

但没有一个集中的文档列出所有已知限制及其原因、影响范围、计划修复时间。新协作者很难快速了解当前模组的边界。

**建议内容**：

```
# 已知限制与设计权衡

按子系统分组，每项包含：
- 限制描述
- 根因（为什么存在这个限制）
- 影响范围（什么场景下会触发）
- 当前状态（待修复/设计特征/上游依赖）
- 如果待修复：计划方案和优先级
```

**推荐位置**：`4-进度与规划/已知限制与设计权衡.md`

**预计工作量**：中（需要从多个文档中提取并分类整理）

#### 2.4 多人/多玩家测试计划

**问题**：多处标注"多人未测试"（轨道摄像机同步、炮塔多人场景、上下车冲突等）。多人场景的 Bug 往往比单人更隐蔽，且难以通过单机调试复现。缺少系统的多人测试计划会导致发布后出现预期外的多人 Bug。

**建议内容**：

```
# 多人游戏测试计划

## 测试场景清单

### 核心交互
- [ ] 两名玩家各自上车不同的载具
- [ ] 一名玩家上车后第二名玩家尝试上车同一载具（应被拒绝）
- [ ] 两名玩家同时上车/下车
- [ ] 一名玩家下车后另一名玩家上车同一载具

### 摄像机
- [ ] 多名玩家各自使用轨道摄像机互不干扰
- [ ] 自适应摄像机在多人场景下正常

### 炮塔与武器
- [ ] 玩家 A 开火，玩家 B 看到弹道渲染
- [ ] 玩家 A 的炮塔追踪，玩家 B 看到炮塔旋转
- [ ] 多人同时开火的网络包压力

### 断线重连
- [ ] 上车后断线重连，状态恢复
- [ ] 驾驶中断线，其他人可再次上车
- [ ] 炮塔配置在断线重连后保持

## 网络包频率与延迟
- 各数据包在 2/5/10 名玩家下的发送频率
- 客户端预测 vs 服务端权威的差异
```

**推荐位置**：`4-进度与规划/多人游戏测试计划.md` 或附加到 `4.1-当前状态.md`

**预计工作量**：小（主要是清单整理，测试执行需多人环境）

### 🟢 低优先级 / 锦上添花

#### 2.9 自动化编译依赖提取脚本

**问题**：`1.2-上手引导与编译.md` 中的 Python 提取命令每次手动复制粘贴容易出错（路径写错、版本号不匹配等）。

**建议**：在项目根目录添加一个 `extract_jars.ps1`（PowerShell）或 `extract_jars.sh` 脚本，一键执行所有提取操作。脚本内容从现有文档中提取即可。

**预计工作量**：小（~30 分钟）

**问题**：`docs/en/` 目前有 7 篇文档，而中文文档有 50+ 篇（含 29 篇关键技术要点）。如果目标是国际社区，关键文档的英文版本缺失会限制海外贡献者的参与。

| 优先级 | 文档 | 原因 |

| ⭐⭐ | `5.1-关键技术要点/` 核心篇 | 编号 3/4/5/7/14/18/25 |
| ⭐ | `2.2/2.3/2.4` 子系统 README | 目录级导航 |

**预计工作量**：大（需逐篇翻译，建议采用"核心先行、逐步覆盖"策略）

#### 2.11 文件名一致性清理

**问题**：`0.1-文档更新规则.md §2.2` 明确禁止文件名中的空格和 `NN. xxx.md` 格式，但以下目录中存在违规文件：

- `2.2-3C控制系统/`：同时存在 `2-上车与下车系统.md` ✅ 和 `2. 上车与下车系统.md` ❌
- 同一目录下：`1-通用驾驶舱（Cockpit）.md` ✅ 和 `1. 通用驾驶舱（Cockpit）.md` ❌
- 其他子系统可能也有类似重复

这些带空格的文件是旧版命名遗留，与规范文件内容重复，可能引起链接混淆和跨平台路径问题。

**建议操作**：
1. 确认每个带空格文件的内容是否与同编号的规范文件完全一致
2. 如果一致，删除带空格版本
3. 检查所有文档中的链接，更新为指向规范文件名
4. 更新对应的 `README.md` 目录

**预计工作量**：小（主要是检查和删除重复）

#### 2.12 性能基准数据补充

**问题**：`5.5-性能分析报告/` 已经有了很好的框架（服务器端负载、客户端负载、网络压力、优化建议 4 篇），但目前缺乏实际的性能测试数据。

```
# 性能基准数据（示例）
```

#### 2.13 设计决策记录（ADR）

**问题**：目前重大设计决策的记录分散在：
- `7-开发日志/` 中按时间线记录（如"为什么放弃抽象齿轮模式"）
- `5.1-关键技术要点/` 中按主题记录（如"为什么不用静态共享 Map"）
- `5.4-设计哲学与路线图/` 中记录战略决策（如"社区资产生态"）

但缺少一个**轻量级的 ADR（架构决策记录）索引**，让新协作者可以快速了解"这个项目为什么长这样"。

**建议内容**：在 `5-技术参考/` 下新增 `架构决策记录索引.md`，列出所有已做出的重要决策：

```markdown
# 架构决策记录（ADR）索引

| # | 决策 | 时间 | 替代方案 | 选择理由 | 详见 |
|:-:|------|:----:|---------|---------|------|
```
