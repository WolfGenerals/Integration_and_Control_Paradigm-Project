# 开发工作流规范

> 面向所有协作者（人类开发者与 AI 智能体）。

---

## 1. 伦理与法务红线

### 1.1 禁止反编译前置项目

**绝对禁止**对本项目依赖的前置模组进行反编译或在项目内生成反编译字节码文件。

前置模组皆为**开源项目**。如果遇到无法从源码找到的 API 或行为，应：
1. 检查 `5-技术参考/5.1-关键技术要点/` — 已有大量踩坑记录和技术解析
2. 检查 `5-技术参考/5.2-踩坑快速索引.md` — 按症状搜索已知问题
3. 联系协作者获取源码指引

### 1.2 前置模组源码获取

| 前置 | 源码地址 |
|:----|:---------|
| Simulated-Project (Sable) | https://github.com/ryanhcode/sable |
| Create | https://github.com/Creators-of-Create/Create |
| NeoForge | https://github.com/neoforged/NeoForge |

也可从本地克隆仓库直接引用。

## 2. 开发环境注意事项

### 2.1 VS Code + Gradle 的已知限制

VS Code 对 Java + Gradle 项目支持不完善，具体表现为：

- Java 文件中大量报错飘红（`The import X cannot be resolved` 等）
- 这是 **IDE 层面的解析缺陷**，并非实际编译错误
- **判断标准**：如果 `gradlew build` 或 `gradlew runClient` 编译通过，飘红可安全忽略

### 2.2 推荐调试方式

- 优先使用**调试代码 + 日志输出**检查逻辑链条
- 使用 `runClient` 任务启动客户端进行实机测试
- **不要**擅自修改 Gradle 构建配置来"修复"飘红

### 2.3 编译依赖提取

首次编译前需从 Create/Simulated JAR 中提取嵌套的 JarJar 依赖到编译 classpath：
- 见 `1-项目介绍/1.2-上手引导与编译.md §编译步骤`

## 3. 文档使用指南

### 3.1 文档定位

文档是快速了解功能、基础设施、代码结构的最优工具，比从头研究源码快得多。

**阅读策略**：
- 不需要一上来全部读完
- 遇到问题时按需查阅 → 使用 `0.2-文档结构总览.md` 的"需求场景"速查表
- 高频查阅路径：踩坑 → `5.2-踩坑快速索引.md` → `5.1-关键技术要点/` 对应编号

### 3.2 文档与实际代码不一致

如果发现文档描述与实际代码行为不符：
1. 以**实际代码行为**为准
2. 记录不一致之处
3. 告知协作者更新文档（遵循 `0.1-文档更新规则.md`）

### 3.3 文学性 vs 技术性

- `5-技术参考/`（技术性）中的内容应视为**一直有效的规则**，除非代码变更
- `7-开发日志/`（文学性）中的内容记录的是**过去的过程**，可能存在过时的技术细节

## 4. 沟通规范

### 4.1 语言选择

- 与当前对话用户保持**相同的自然语言**沟通
- 例如：用户说中文，就用中文思考和回复；用户说英文，就用英文
- 不受文档本身语言的限制

### 4.2 修改文档前的必读

在修改任何文档文件之前，**必须先阅读**：
1. `0-文档管理指南/0.1-文档更新规则.md` — 文档维护规范与单一事实来源原则
2. `0-文档管理指南/0.2-文档结构总览.md` — 了解当前结构和各文档职责
