# 文档更新规则

---
updated: 2026-06-13
status: current
maintainer: @项目协作者
---

> 本文档是项目文档管理的"元规则"。所有协作者在修改文档前，**必须**先阅读本文。
> 违反规则的文档修改可能会被驳回。

---

## 1. 核心原则

### 1.1 文档是代码的"地图"，不是代码的"副本"
- 文档描述**设计决策、架构原理、踩坑教训**，而不是逐行罗列代码做了什么。
- 如果某信息能从代码（注释、类名、方法签名）直接推断，就不需要写入文档。

### 1.2 消除重复，单一事实来源
- **每条信息只在一个地方维护**，其他地方通过链接引用。
- 例如：功能状态表格 → `2.1-功能总表.md`；当前活跃问题 → `4.1-当前状态.md§活跃问题`。
- ✅ `详见 2.1-功能总表.md §悬挂与载具控制`
- ❌ 在 `4.1` 中再写一遍 "悬挂测试方块 ✅ 06-01 完成"

### 1.3 文学性与技术性分离
文档分为两大类：

| 类别 | 内容 | 位置 | 时效 |
|:----|:----|:----|:----|
| **文学性（过程）** | "为什么"、"发生了什么"、时间线 | `7-开发日志/` | 过去时，写完后不修改 |
| **技术性（规则）** | "这是什么"、"不要做什么"、一直有效的知识 | `5-技术参考/` | 现在时/将来时，随代码同步更新 |

- 技术文档（`5-技术参考/`）中**不要**出现过程性叙述（"今天发现了..."），只写结论和规则。
- 开发日志（`7-开发日志/`）中涉及技术细节时，应引用 `5-技术参考/` 中的对应文档，而非重复撰写。
- 旧日志不删、不改，保留在 `7-开发日志/` 中供追溯。

### 1.4 变更时同步更新
- 代码变更影响某个功能的行为时，对应功能文档必须同步更新。
- 新增功能时：更新 `2.1-功能总表.md` + 对应子系统的文档 + 可选的 `5.1-关键技术要点`。
- 修复 Bug 时：在 `5.2-踩坑快速索引.md` 和对应 `5.1-关键技术要点` 文档中添加条目。

## 2. 文档文件标准

### 2.1 文件头格式

每个文档文件**必须**以如下元数据块开头（本次审查已批量补齐）：

```markdown
---
updated: 2026-06-13
status: current        # current | needs-review | archived
maintainer: @协作者名
---
```

- `status: current` — 内容与代码一致，可放心阅读
- `status: needs-review` — 可能已过时，阅读时需谨慎
- `status: archived` — 保留供历史参考，不再更新

### 2.2 文件命名规范

```
NN-短横线分隔的中文标题.md
```

- 编号保持两位（如 `01`, `02`... `29`）
- 使用有意义的短名称，不超过 30 字
- **禁止**在文件名中使用空格。用短横线 `-` 替代，如 `4-摩擦圆物理.md` ❌ `4. 摩擦圆物理.md`
- **禁止**在文件名首部使用 `.` 加空格（`NN. xxx.md`），统一用 `NN-xxx.md`
- 子文件夹内统一编号风格
- 包含特殊字符（括号、`@`、`:` 等）的文件名需确保跨平台可访问

### 2.3 目录/README 结构

每个子文件夹必须有 `README.md`，格式：

```markdown
# 文件夹名称

一句话说明本文件夹的用途。

| 编号 | 文件 | 一句话说明 |
|:----:|------|-----------|
| 1 | [文件名](文件名.md) | 用途简述，≤ 20 字 |
| 2 | ... | ... |
```

## 3. 各文件夹维护规则

### 3.1 `1-项目介绍/`
- **何时更新**：依赖关系变更、编译步骤变化时
- **频率**：低（通常不变）

### 3.2 `2-功能清单/`
- **`2.1-功能总表.md`** — **单一事实来源**。所有功能需在此登记，含状态、一句话说明、链接。
- **子系统文档**（`2.2/` `2.3/` `2.4/`）— 描述功能的**设计原理和使用方式**，而不是代码实现细节。代码实现细节 → `5-技术参考/5.1-关键技术要点/`
- **何时更新**：新增功能、功能行为变更时
- **编号检查**：新增条目后检查编号连续无冲突，各小节计数与实际行数一致
- **分类检查**：每个功能归属正确的子系统，不应出现跨子系统重复登记（如 WASD 智能映射不应同时出现在悬挂和 3C 中）

### 3.3 `3-架构与文件结构/`
- **`3.1-SubLevel物理架构.md`** — 核心概念，只更新在架构级变更时
- **`3.3-代码文件索引.md`** — 新增/删除/重命名 Java 文件时必须同步更新

### 3.4 `4-进度与规划/`
- **`4.1-当前状态.md`** 仅维护：
  - ⚠️ 当前活跃问题
  - 🧊 已暂停项目
  - ✅ 已验证的代码行为
  - ❌ **不**包含已完成功能的逐项清单（已在 `2.1-功能总表.md`）
  - ❌ **不**包含待办功能列表（已迁移至 `5.4-设计哲学与路线图/7-待办路线图.md`）
- 待办功能统一在 `5.4-设计哲学与路线图/7-待办路线图.md` 维护，`4.1` 仅通过链接引用
- **`7-开发日志/推进周期/`** — 开发周期记录（已从 `4.2` 迁入），完成后**不**再修改

### 3.5 `5-技术参考/`
- **`5.1-关键技术要点/`** — 每个踩坑/技术决策一篇，格式：

  ```markdown
  # 标题

  ## 背景
  一句话说明遇到的问题。

  ## 根因
  一句话说明根本原因。

  ## 解决方案
  代码级的解决方案描述。可包含关键代码片段（≤ 20 行）。

  ## 参考
  - 相关文件链接
  - 相关历史记录
  ```

- **`5.2-踩坑快速索引.md`** — 按症状索引，每条 ≤ 3 行，链接到 `5.1` 的详细文档
- **新增规则**：新建 `5.1` 文档时，必须同时在 `5.2-踩坑快速索引.md` 和 `5.1/README.md` 中添加索引

### 3.6 `6-愿景/`, `7-开发日志/`
- **文学性资产**，记录"为什么"和时间线。
- 写完后**不**修改已有文件（保持历史原貌）。
- 新的开发记录直接追加新文件即可。
- 涉及技术细节时引用 `5-技术参考/`。

## 4. 文档更新工作流

### 4.1 小更新（修复错别字、更新日期、补充细节）
- 直接编辑对应文件，无需审批。

### 4.2 中更新（新增功能文档、修正技术要点）
1. 更新 `2.1-功能总表.md` 添加新功能行（注意编号连续 + 分类正确）
2. 在对应子系统的 `README.md` 和文档中添加说明
3. 如有踩坑/技术决策 → 创建 `5.1` 文档 + 更新 `5.2` 索引
4. 如涉及代码文件结构 → 更新 `3.3-代码文件索引.md`
5. 如新增 Java 文件 → 确认 `3.3-代码文件索引.md` 已添加对应行
6. 如涉及运行时/编译时常量 → 引用时注明全限定类名（`Config.java` vs `IACPConfig.java`）
7. 更新日期同步：修改文档时同步更新文件头 `updated` 字段

### 4.3 大更新（架构变更、文档重构）
- 需通知协作者，确认变更方案后再执行。
- 完成后更新 `README.md` 的导航结构。

## 5. 文档质量检查清单

提交文档修改前，请自查：

- [ ] 是否有信息重复了其他地方已存在的内容？
- [ ] 是否更新了所有相关的索引文件？（`2.1-功能总表`, `3.3-代码文件索引`, `5.2-踩坑快速索引`）
- [ ] 文件头是否包含 `updated` 日期？
- [ ] 代码片段是否与最新代码一致？（如果引用了常量，确认常量名未变更）
- [ ] 新增的文件是否在对应的 `README.md` 中登记了？
- [ ] 链接是否可访问？（相对路径是否正确？）
- [ ] `2.1-功能总表.md` 各节计数与实际行数一致？编号连续无冲突？
- [ ] 功能归属子系统是否正确？（不跨子系统重复登记）
- [ ] 引用的 Java 类名是否使用全限定类名以消除歧义？（`Config.java` 运行时 vs `IACPConfig.java` 编译时）
- [ ] 代码现有功能是否已在文档中体现？（新增功能应同时新增文档，避免遗漏）

## 6. 工具与自动化

### 6.1 文档健康检查
可定期运行以下检查：
- `2.1-功能总表.md` 中所有功能的"详见"链接是否有效
- `2.1-功能总表.md` 各节计数与行数是否一致、编号是否连续
- `3.3-代码文件索引.md` 中的 Java 文件是否存在
- `5.2-踩坑快速索引.md` 中的"详见"链接是否有效
- 所有文档是否包含文件头（`--- updated: ... ---`）
- 文件名是否符合 `NN-短横线分隔.md` 规范（无空格）

### 6.2 模板速查
| 场景 | 模板/参考 |
|------|-----------|
| 新增功能 | `2.1-功能总表.md` 添加一行（注意编号+分类）+ 子系统文档 |
| 新增踩坑 | `5.1/` 新文档 + `5.2-踩坑快速索引.md` 添加行 |
| 新增代码文件 | `3.3-代码文件索引.md` 加一行 |
| 新增周期记录 | `7-开发日志/` 新文件 |
| 引用 Config 类 | 注明 `Config.java`（运行时`ModConfigSpec`）或 `IACPConfig.java`（编译时常量） |
