## 第二阶段：方向机控制迭代 ~25 轮

| # | 方案 | 结果 | 根因 |
|:-:|------|:----:|------|
| 1 | `applyTorqueImpulse` | ❌ 撕裂载具 | 早期约束未就绪 |
| 2-4 | `setMotor` PD (SmartBlockEntity) | ❌ 转一次即停 | SmartBlockEntity 无 RPM 感知 |
| 5-7 | `addLinearAndAngularVelocity` | ❌ 不旋转 | forward 方向 (0,1,0)=UP |
| 8-10 | 速度伺服 + SNAP teleport | ❌ 到位不稳 | solver substep 过冲 |
| **11-20** | **抽象齿轮模式** | ❌ 震荡 | 见下 |

### 抽象齿轮模式（废弃）

**设计**：齿轮永不断开，旋转时高 RPM + 高 kP=2000/kD=160 驱动，停止时 0RPM 堵转制动。

**变更**：
1. `TurretBaseBlock`：`Block` → `KineticBlock + ICogWheel`
2. `TurretBaseBlockEntity`：`SmartBlockEntity` → `KineticBlockEntity`
3. 方向取反：`dir = -(int)Math.signum(yawErr)`
4. 恒定 kP=2000/kD=160

**问题**：方向正确、车体安全，但在目标附近左右摇摆 ❌

## 第三阶段：诊断 —— DebugGear 与 RPM 阶跃（关键转折）

### 调试齿轮 (DebugGearBlock)
- 小型齿轮，`RotatedPillarKineticBlock + ICogWheel`
- N 键切换调试输出，每 tick 打印 `[DebugGear] speed/angular/RPM`
- `DebugGearToggleC2SPacket` 服务端切换

### ⭐ 关键发现：Create RPM 是阶跃变化

实测 45 tick 窗口：
```
tick=7509 speed=+0.0  ← 静止
tick=7510 speed=-16.0 ← 第1 tick 突然 -16
tick=7526 speed=-88.0 ← 第17 tick 突然 -88
tick=7554 speed=+0.0  ← 下1 tick 突然归零
```

**结论**：Create 应力网络 RPM 变化是**瞬时的**（阶跃），无任何渐变或加减速过程。这意味着：
- 抽象齿轮模式中"角度累积（开环）+ PD 伺服（闭环）"= **双重积分器系统** → 必然震荡
- RPM 从 ±80 → 0 的瞬间，累积角度停止但物理惯性导致过冲 → PD 回调 → 又过冲

## 第四阶段：最终方案 —— 位置模式 PD 伺服

### 核心洞察

放弃"RPM → 角度累积"的开环思路，参考 Create 的 `SwivelBearingBlockEntity` 使用**纯位置模式 PD 伺服**。

### 架构

```
AimController (每 server tick):
  └─ 从当前 pose + yawErr 算出绝对目标角度
  └─ tb.setTargetYawAbsolute(degrees)

BE.updateYawServo() (每 tick):
  └─ angleLerp(1.0, lastTarget, target) partialTick 插值
  └─ setMotor(DEFAULT_AXIS, goal, kP=200, kD=16, false, 0)
     └─ position mode (false)
```

### 关键代码

```java
// AimController — 绝对角度
double currentYawRad = Math.atan2(fnx, fnz);
double offsetRad = Math.toRadians(Config.TURRET_YAW_OFFSET.get());
double targetYawRad = -(currentYawRad + yawErr) + offsetRad;  // 取反 + 校准
tb.setTargetYawAbsolute(Math.toDegrees(targetYawRad));

// BE — 位置伺服（参考 SwivelBearing）
swivelBearingHandle.setMotor(DEFAULT_AXIS, goal, 200.0, 16.0, false, 0.0);
```

### 坐标镜像（atan2 取反）

`atan2(cross, dot)` 以逆时针为正，而 RotaryConstraint.setMotor 以顺时针为正。因此：
```java
targetYawRad = -(currentYawRad + yawErr) + offsetRad;
//               ^ 取反符号
```

测试验证：0°/180° 方向正确，90°/270° 也正确（不再镜像）。

### 最终参数

| 参数 | 值 | 说明 |
|------|:---:|------|
| SERVO_STIFFNESS | 200.0 | PD 刚度 |
| SERVO_DAMPING | 16.0 | PD 阻尼 |
| YAW_GAIN | 80.0 | 比例增益 |
| YAW_DEAD | 0.5° | 死区 |
| PITCH_DEAD | 0.5° | 俯仰死区 |
| PITCH_CLAMP | 60° | 俯仰最大角 |
| TURRET_YAW_OFFSET | configurable | 偏航校准（-180~180°）|
| MAX_RAY_DISTANCE | 1000 格 | 射线最大距离 |

## 关键教训

1. **Create RPM 是阶跃，不是渐变**：应力网络 RPM 变化在 1 tick 内完成，不能用"虚拟 RPM + 角度累积"的开环控制
2. **位置模式 PD 伺服才是正解**：参考 SwivelBearing，每 tick position-mode setMotor，不依赖 RPM
3. **SmartBlockEntity ≠ KineticBlockEntity**：前者无 RPM 感知，setMotor 后续调用不响应
4. **atan2 符号与约束符号相反**：取反解决方向镜像问题
5. **诊断工具至关重要**：DebugGear 直接暴露了 Create RPM 的本质行为，消除了所有猜想

## 配置校准

在 `config/iac_p-common.toml` 中：
```toml
[turret.aim]
    # 炮塔方向机偏航角度校准偏移（单位：度）
    # 正值使炮塔顺时针偏转，负值逆时针
    yawOffset = 0.0
```
