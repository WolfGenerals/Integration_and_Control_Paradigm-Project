# 2026-06-14 至 06-15：瞄准系统全面重构

## 目标

彻底重构炮塔瞄准系统，消除从鼠标移动到炮塔开始旋转之间的每一毫秒空耗时间，同时将瞄准方式从"发坐标→算角度"改为"直接发角度+视差修正"。

## 过程

### 第一轮：链路分析（06-14）

分析瞄准全链路各阶段耗时分布：

| 阶段 | 内容 | 最差耗时 |
|:----|------|:--------:|
| ① | 等 client tick 边界 | 50ms |
| ② | performRaycast SubLevel 全遍历 | 5ms |
| ③ | 网络传输 | 0-100ms |
| ④ | 等 server tick 边界 | 50ms |
| ⑤ | Chunk 扫描炮塔底座 | 5ms |
| ⑥ | AimController 比例增量计算 | 0.5ms |
| **合计** | | **~211ms** |

**关键发现**：
- **双重限速**：AimController 的 `YAW_MAX_STEP_DEG=3.0°/tick` + BE 的 `yawSpeedPerTick()=4.5°/tick`，有效速度取小值 60°/s
- **server tick 边界等待**：`TurretTargetC2SPacket` 只把目标存到 TARGETS map，真正计算要等 `ServerTickEvent.Post`

### 第二轮：Packet handler 直接驱动 + ComponentRegistry

改动 `TurretTargetC2SPacket.handle()`：
- 优先用 `ComponentRegistry.getComponents(vehicleUUID, TURRET_BASE)` O(1) 查找炮塔底座
- 直接调用角度计算并设目标，跳过 AimController TARGETS map
- 消除 chunk 扫描 (1~5ms) + server tick 边界等待 (0~50ms)

### 第三轮：角度直驱（06-15）

用户提出：不再发命中坐标，直接发摄像机朝向角度，服务端减去载具偏航即可。

- `TurretTargetC2SPacket` record 从 `(double x, double y, double z)` 改为 `(double cameraYaw, double cameraPitch)`，StreamCodec 从 3 double 改为 2 float
- 服务端：`turretYaw = cameraYaw - vehicleYaw`，`turretPitch = -cameraPitch`
- 去除 `TurretAimController` 中的 TARGETS map、setTarget()、clearTarget()、tick()、drive()、drivePitch()
- 新增 `driveAnglesImmediate(tb, targetYaw, targetPitch)` — 直接设目标，零运算

### 第四轮：消除限速轨迹规划

用户指出 `setTargetYawAbsolute()` 中每 tick `min(|delta|, yawSpeedPerTick)` 的限速是多余的——PD 伺服本身就是平滑的位置控制器。

- 改为直接覆写：`this.targetAngleDegrees = degrees`
- `Config.TURRET_YAW_SPEED_DPS` 和 `Config.TURRET_PITCH_SPEED_DPS` 不再生效

### 第五轮：PD 阻尼调参

- 方向机 `SERVO_DAMPING` 50→0（用户测试，纯 P 控制过冲严重）
- 最终调到 20，平衡响应速度和过冲抑制

### 第六轮：视差修正

用户提出武器和摄像机位置不同导致近距瞄准偏差的问题，用几何方案解决：

- 在 `ClientEvents.onClientTick()` 中，每 tick 获取命中点 O 和炮塔位置 C
- 客户端本地计算炮塔→O 的方向角（`atan2`/`asin`）
- 直接作为修正后的目标角度发送，不需要将命中坐标传到服务端
- 角度符号修正：`atan2` 输出数学 CCW+，取反转为 MC CW+ 惯例

### 第七轮：发命中坐标 + 真三维球坐标 + 载具局部空间计算（06-15 终版）

前几轮的「发角度+视差修正」方案存在两个根本问题：
1. 视差修正使用**载具中心而非各炮塔自身位置** → 多炮塔平行瞄准无汇聚
2. 载具翻转时世界空间角度计算错误

**数据格式再反转**：`TurretTargetC2SPacket` 从 `(cameraYaw, cameraPitch, hitRange)` 改回 `(hitX, hitY, hitZ)`——发送命中点世界坐标。
之所以不再担心「坐标→角度 atan2 延迟」，因为之前的延迟优化（ComponentRegistry O(1) + 直接 handler）已经消除了真正的瓶颈，atan2 开销微不足道。

**核心设计——载具局部空间三维球坐标**：
- 所有坐标通过 `vPose⁻¹ · (world - vPos)` 变换到载具局部空间
- **方向机 Yaw（俯视投影 / XZ 平面）**：`-atan2(dx, dz)` — 天然是载具相对角度，无需额外减偏航
- **高低机 Pitch（侧面投影 / 垂直面）**：`atan2(dy, sqrt(dx²+dz²))` — 在包含炮管的垂直面内计算
- 每座炮塔从自身砂轮 SubLevel 位置出发 → 所有炮塔瞄准同一点 → **真实火力交叉**
- 载具翻转时局部「水平面」跟随载具，角度始终正确

**核心改进——`driveImmediate()` 消除 1-tick 空等**：
- 原：`setTargetYawAbsolute()` 只设字段 → 等下一 tick `BE.tick()` → `updateYawServo()` → `setMotor()`
- 现：`driveImmediate()` 设目标后**立即执行 `updateYawServo()` + `setMotor()`**，消除 0~50ms 空等
- `tick()` 中的冗余调用因 `lastTarget == target` 无副作用

### 第八轮：准星偏向锥角（延迟掩饰）

在 `WeaponOverlay.fireSingleTurret()` 中增加**准星偏向锥角**机制：
- 对射线方向施加最大 ±1° 的锥角偏移，偏向当前准星瞄准点
- 若炮塔偏差 ≤ 1°，弹道完全对准准星（理想命中）
- 若偏差 > 1°，限制到锥角边界（最大补偿 1°）
- 实质是**客户端弹道预测（client-side hit prediction）**，掩饰炮塔旋转延迟 + 网络 RTT

### 延迟优化总结

| 阶段 | 内容 | 优化前 | 优化后 |
|:----|------|:----:|:-----:|
| ① | 等 client tick 边界 | 50ms | **50ms**（不可压缩） |
| ② | 射线检测 + 发包 | ~0.1ms | ~0.1ms |
| ③ | 网络传输 | 0~100ms | 0~100ms（物理极限） |
| ④ | 服务端查找炮塔底座 | chunk 扫描 1~5ms | **ComponentRegistry O(1) ~0ms** |
| ⑤ | server tick 边界 | TARGETS map 等 ServerTickEvent 0~50ms | **直接 handler 0ms** |
| ⑥ | 角度计算 | atan2 坐标→角度 ~0.002ms | ~0.002ms（不变） |
| ⑦ | setTarget → setMotor | 等下一 tick 0~50ms | **driveImmediate 立即执行 0ms** |
| **合计** | | **~211ms** | **~51ms** |

另加准星偏向锥角 ±1° 作为延迟掩饰，感知延迟进一步降低。

### 剩余问题

- 准星偏向锥角在服务端独立验证时可能因客户端-服务端命中位置差异导致微量不一致（±1° 内，可忽略）
- 回退路径（chunk 扫描）无 SubLevel pose 上下文，用地毯位置近似炮塔位置，精度略低
