# 1. 战略罗盘与设计哲学

## 命名由来

**中控载具工坊** —— 三重心意：

| 字面 | 含义 |
|------|------|
| **中控** | 中央控制（C 键调参 GUI，中央控制系统） |
| **中控** | Minecraft 中的控制（模组核心功能） |
| **载具工坊** | 玩家造车、调车、开车的平台 |

## 🧭 战略罗盘

| # | 原则 | 说人话 | 反面（警惕） |
|---|------|--------|-------------|
| 1 | **代码做胶水，不造轮子** | 用 Create 的机械、Sable 的物理、MC 原版的渲染做地基，我们只写连接它们的"胶水代码" | 自己从零写物理引擎/渲染管线/网络库 |
| 2 | **美术占位符主义** | 纹理用原版 MC + Create 现成的，音效用 MC 原版，特效？不需要。这就够了 | 花钱雇美工 / 花时间雕贴图 / 买音效包 |
| 3 | **社区资产生态，非官方素材包** | 做素材包导入框架，让社区自己画纹理、做音效、写特效 | 官方出付费材质包 / 官方雇人做音效 |
| 4 | **玩法深度 > 视觉 polish** | 玩家的"哇"应该来自物理反馈、调参深度、战术可能性 | 花 3 天调一个粒子效果，花 3 小时写一个物理规则 |
| 5 | **不做服务型游戏** | 没有 KPI、没有月活指标、没有付费转化率 | 开始考虑"怎么吸引更多玩家"、"怎么变现" |
| 6 | **渐进式复杂度** | 新手 10 秒开走，老手 10 年调不完 | 上来就甩给玩家 50 个滑块 / 或者把所有参数都藏起来 |
| 7 | **拥抱方块美学** | 这是 Minecraft，不是虚幻 5 | 试图让 MC 看起来像 Crossout / 用光影包当卖点 |
| 8 | **问心无愧** | 每一行代码都是因为想写，不是因为有人要你写 | 为了"交付"而砍功能 / 为了"赶进度"而堆屎山 |

## 核心设计哲学

### 一句话卖点
> **在 Minecraft 里造一辆物理真实的载具，调悬挂、调齿比、调轮胎气压，然后开出去征战。**

### 渐进式复杂度揭示
```
让新手 10 秒就能开上车，让老手 10 年都调不完。
```

### 两层架构
```
┌─────────────────────────────────────────────────────┐
│              上层体验（预设配置层）                     │
│  简单模式（默认）         高级模式                      │
│  ├─ 齿比预设              ├─ 全量齿比定制               │
│  ├─ 驱动力预设            ├─ 发动机调参 GUI             │
│  ├─ 悬挂软/中/硬          ├─ 悬挂全参数                 │
│  ├─ 轮胎默认              ├─ 轮胎气压/宽度/扁平比       │
│  └─ 方块上限 128          └─ 可关闭上限                 │
│  新手 10 秒开走           老手 10 年调不完              │
├─────────────────────────────────────────────────────┤
│              底层基础（物理引擎层）                     │
└─────────────────────────────────────────────────────┘
```
