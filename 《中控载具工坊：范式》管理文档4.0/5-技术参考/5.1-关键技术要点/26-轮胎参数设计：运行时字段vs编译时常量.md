# 26. 轮胎参数设计：运行时字段 vs 编译时常量

### 核心教训
**不要将调试期间暴露的运行时参数当作正式功能。** 早期为方便调参将 7 个轮胎物理参数全部做成 NBT 持久化的运行时字段，但普通玩家不需要也不应该面对胎体刚度/胎内容积等参数。

### 最终设计原则

```
轮胎款式决定所有物理属性（半径/胎宽/刚度/滚阻/摩擦系数等）
                          ↓
            玩家唯一可调 = 胎压（打多少气）
```

### 代码演进路线

| 阶段 | 状态 | 说明 |
|:----:|:----:|------|
| 初期 | 7 个运行时 NBT 字段 | treadWidth、carcassStiffness、maxPressure、nominalPressure、tireVolume、crrBase、crrDeformationGain，全部在 C 键菜单可编辑 |
| **06-09** | **精简为仅胎压** | 6 个字段→编译时常量（DEFAULT_*），仅保留 `nominalPressure` 为运行时字段；简化 TireConfigC2SPacket 和配置界面 |
| 未来 | TireType 注册表 | 每种轮胎款式（越野/赛车/重载）内置完整物理规格，玩家只需选胎和打气 |

### 决策检查清单

当考虑是否将某个参数做成运行时 NBT 字段时，问：

1. **玩家真的需要调节这个参数吗？** 胎压 → 是。胎体刚度 → 一般不是。
2. **这个参数因轮胎款式而异吗？** 是 → 应该编码进轮胎类型定义，而非运行时字段。
3. **NBT 持久化+网络同步的复杂度值得吗？** 每个运行时字段 = TAG 常量 + read/write + 网络包 Codec + 兼容性检查。
4. **有没有可能用编译时常量替代？** 先做常量，等确实需要运行时调节时再升级。
